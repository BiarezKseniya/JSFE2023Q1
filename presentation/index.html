Key Concepts of XSS

  • XSS: a web-based attack on vulnerable web apps.
  • The victim is the user and not the application.
  • Malicious content is delivered to users using JS.
XSS is an attack where malicious code is injected into safe websites. An attacker exploits a vulnerability in a web application to send code, usually JavaScript, to an end user. XSS attacks target the application’s users directly, rather than the application itself. In short, XSS vulnerabilities occur when input coming into web applications is not validated and/or output to the browser is not properly escaped before being displayed.

Types of XSS attacks

  • Persistent
  • Reflected
  • DOM-based
There are three main types of XSS attacks: persistent, reflected, and DOM-based.

Persistent XSS

Expected behaviour:


						<p>Hello, this is my message!</p>

Evil hacker:


						<p><script>/* Bad stuff here... */</script></p>
Occur when malicious code is stored on a server and shown to other users. Ex: a message on a forum, a nickname in a chat room, or contact details in an order. The main idea is that an attacker can send data that attacks other users. Ex: a message board application lets users submit messages, which are displayed to other users. If the application doesn't perform any other processing of the data, an attacker can easily send a message that attacks other users:

Reflected XSS

Expected behaviour:

https://insecure-website.com/status?message=All+is+well.


						<p>Status: All is well.</p>
On the other hand, reflected cross-site scripting involves the immediate return of user input. An attacker must trick the user into sending data to the target site, which is often done by tricking the user into clicking a maliciously crafted link. It exploits the situation when an application receives data from an HTTP request and returns it in the response without validation.

Reflected XSS

Evil hacker:

https://insecure-website.com/status?message= <script>/+Bad+stuff+here…+/</script> 


						<p>Status:
							<script>/* Bad stuff here… */</script>
						</p>
Ex: If a user visits this URL, the attacker’s script will execute in their browser. Generally, search results and error message pages are two common targets for reflected XSS. They often send unmodified user input as part of the response without ensuring that the data is properly escaped so that it is displayed safely in the browser.

DOM-based

Expected behaviour:


					const search = document.getElementById('search').value;
					const results = document.getElementById('results');
					results.innerHTML = 'You searched for: ' + search;

Evil hacker:


						<p>You searched for:
							<img src=1 onerror='/* Bad stuff here... */'></p>
Occur when malicious code changes the structure of a page in the user’s browser. Ex: an app uses some JS to read the value from an input field and write that value to an element within the HTML. If the attacker can control the value of the input field, they can easily construct a malicious value that causes their own script to execute.

XSS Impact

XSS vulnerabilities are one of the OWASP Top 10 security concerns today.

XSS attacks can have serious consequences. They are even considered to be one of the top security issues by OWASP, especially as so many organizations rely heavily on web applications for customer interaction and validation.

Consequences

  • Account hijacking
  • Spreading web worms
  • Accessing browser history and clipboard contents
  • Remotely controlling the browser
  • Scanning and exploiting intranet appliances and apps
When attackers succeed in exploiting XSS vulnerabilities, they can gain access to account credentials. They can also spread web worms or access the user’s computer and view the user’s browser history or control the browser remotely. After gaining control to the victim’s system, attackers can also analyze and use other intranet applications.

Good Practises

  • Filter input on arrival
  • Encode data on output
  • Use appropriate response headers
  • Content Security Policy
XSS vulnerabilities can be prevented by consistently using secure coding practices and ensuring that all input is properly escaped. At the point where user input is received, we can filter as strictly as possible based on what is expected or valid input. At the point where user-controllable data is output in HTTP responses, we can encode the output to prevent it from being interpreted as active content. Depending on the output context, this might require applying combinations of HTML, URL, JavaScript, and CSS encoding. To prevent XSS in HTTP responses that aren't intended to contain any HTML or JavaScript, you can use the Content-Type and X-Content-Type-Options headers to ensure that browsers interpret the responses in the way you intend. • Also, you can use Content Security Policy (CSP). CSP is a browser mechanism which has a standardized collection of directives to instruct the browser which content sources can be trusted and which should be prevented. However, the CSP can easily be bypassed if it is not strict enough. Thus, CSP should be used as a last line of defense to reduce the severity of any XSS vulnerabilities that still occur. XSS flaws can be difficult to identify and remove from a web application. The best way to find flaws is to perform a security review of the code and search for all places where input from an HTTP request could possibly make its way into the HTML output. All in all, although XSS vulnerabilities are a major security concern, they can be prevented by writing secure code, testing for vulnerabilities, and using security tools.
